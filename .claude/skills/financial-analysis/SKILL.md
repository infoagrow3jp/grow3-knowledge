# financial-analysis：決算書分析の実行手順

## 正本の位置づけ（複製禁止）

- **指標の定義・算式・閾値・区分名称の正本**：
  `docs/financial-analysis_指標定義書_v0.2.md`
- **計算の実装（正本の実行形）**：`financial_calc.py`
- **本スキルが持つもの**：上記2つをどの順序・どの確認手順で使うかという
  手順のみ。指標の算式・閾値・区分名称を本ファイルに書き写さない
  （書き写した瞬間に、定義書の改訂とスキルの記載がズレる二重管理リスクが
  生じるため）。指標の意味・算式・閾値を確認する必要が生じたら、その場で
  定義書を読みに行く。
- 判定区分名称は定義書の確定名称を**一字一句そのまま**使用する
  （DEC-004。「危険」「良好」等の一般的なレーティング語に言い換えない）。

## 全体構成（判定の3層構造）

レポートは指標定義書§18「判定の3層構造」に従い、次の順序で組み立てる
（各層の算式・閾値は定義書を参照。ここでは順序と役割のみ記す）：

1. **第1層：フロー — CF自走性**（§18・第0指標）：キャッシュフローは
   自走しているか。税務損益より優先して見る。
2. **第2層：ストック — 債務償還年数**（§15・§2）：借金の重さは
   持続可能か。
3. **第3層：バッファ — 余剰現預金**（§17）：突発的な支出への
   ショック耐性はあるか。

これに加え、レポートの軸は**中核4指標**（CF自走性／債務償還年数／
限界利益率／労働分配率、§14）に絞る。多指標ダッシュボードにしない。
自己資本比率・ROE等のB/S系指標は中核4指標と並列表示せず、**参考欄で
「気づきのトリガー」**として扱う（相場から大きく乖離した場合にのみ
注意を立てる。合否判定には使わない。§14参照）。設計思想「財務統計は
積み上げであり他社との横比較に意味は薄い。見るのは企業の持続可能性
のみ」（§14）を報告の軸に置く。

## 実行手順

### Step 0：実行前チェック（financial_calc.py実行前に必須）

次を確認し、確認結果を記録する。不足がある場合でも直ちに停止せず、
不足に応じて指標ごとに「計算可能／参考値扱い／算定不能」に分類する
（§10 工程3段のStep1「入力検証」に対応）。

- 対象期間（何期分か・決算期はいつか）
- 単位（円／千円／百万円）
- P/L・B/S・C/Fそれぞれの有無（C/F計算書が無い場合は営業CFが
  推計層または簡易層になる。§1-1参照）
- 前期比較データ（前期末B/S）の有無（無いと推計営業CFが算出できず、
  簡易層にフォールバックする。§1-1-4参照）
- 維持投資の実額（設備投資支出額）の有無（無いと減価償却費が代理値
  になる。capex_source=depreciation_proxy）

この時点で§10の停止条件（B/S貸借不一致・前期末と当期首の不一致等）
に該当する資料上の異常が見つかった場合は、見立てに進まず資料の
確認を先に依頼する。

### Step 1：入力データの整形

financial_calc.pyの各関数（`analyze_period()`・`calc_cf_self_sufficiency()`
等）のシグネチャ・docstringに合わせて入力を整形する。本スキルはキー名を
複製しないため、実装時は`financial_calc.py`を直接読んで確認する。

整形時の必須ルール（§1-1-5・実装の安全動作）：
- **フィールドが存在しない、または`null`の場合のみ「欠落」として扱う**
  （その項目を渡さない、またはNoneを渡す）。
- **数値としての`0`は有効な入力値**として渡す（欠落と混同して省略しない。
  例：仕入債務の増減が実際にゼロだった期でも`0.0`を明示的に渡す）。

### Step 2：financial_calc.pyの実行

Shell等でPythonを実行し、`financial_calc`をimportして該当関数を呼ぶ。
暗算・手計算で代用しない（§0-1暗算禁止）。実行結果（入力値・使用式・
出力値）を保存する。

### Step 3：出力メタデータの検証（レポート作成前に必須）

計算結果に次のメタデータが含まれているかを検証する。**1つでも欠落して
いる場合はレポート作成に進まず、欠落しているメタデータと原因を報告して
停止する**（暗算・手計算での埋め合わせを禁止する§0-1の趣旨をレポート
工程にも適用する）。

- `ocf_source`（`actual` / `estimated` / `simplified`）
- `capex_source`（`actual` / `depreciation_proxy`。CF自走性を算定した場合）
- `judgment_status`（`formal` / `screening_only`。CF自走性を算定した場合）
- `warning` ／ `warning_code`（該当する場合。無ければNoneであることを確認）
- 計算過程（`calculation_trace`。使用した費目・式を含むこと）

### Step 4：judgment_statusに応じたレポート表示の分岐

- **`judgment_status == "formal"`**：定義書の確定区分名称をそのまま表示する
  （DEC-004）。算式・使用した営業CFの層（`ocf_source`）を併記する
  （§0-2算式併記義務）。
- **`judgment_status == "screening_only"`**：
  - **正式な判定区分（ゾーン名）を表示しない**（`judgment`はNoneのまま
    表示し、「標準」「余裕」等の区分語を割り当てない）。
  - 参考計算値（`cf_self_sufficiency`は無いが、簡易営業CFの参考値は
    出す）と計算過程は表示する。
  - `warning`の理由文をそのまま表示し、`warning_code`が
    `SIMPLIFIED_OCF_NOT_ELIGIBLE_FOR_FORMAL_JUDGMENT`の場合は
    「2期分のB/SまたはCF計算書が無いため正式判定不可」の趣旨を明示する。
  - 運転資本変動の大小を判断できる情報（前期比の売掛金・棚卸資産の
    急増等）があれば、参考値と実態の乖離リスクについても付記する。

### Step 5：BASTベンチマーク参照（労働分配率をBAST定義で比較する場合のみ）

参照ルールの正本は指標定義書§6-2・§16。本スキルでの手順は次のとおり：

1. **都度Web参照**：WebSearch／WebFetchでTKC経営指標（BAST）速報版
   （https://www.tkc.jp/tkcnf/bast/sample/ ）を毎回参照する。過去の
   セッションで取得した値・記憶上の値で補完しない。
2. **取得できた場合**：版（例：令和◯年版）・対象決算期・業種・
   企業区分（該当があれば）・参照日・参照元URLを記録し、レポートの
   出典欄に明記する。
3. **取得できない場合**（アクセス不可・該当業種の値が無い等）：
   数値を創作・記憶で補完せず**「取得不能」と明示**する。この場合も
   自社指標の算定・内部判定（financial_calc.pyの判定区分）は
   変更せずそのまま継続する（外部ベンチマークの欠落を内部判定の
   停止条件にしない）。
4. **保存禁止**：取得したBASTの数表・数値をリポジトリ・データファイルに
   保存・蓄積しない（§6-2）。レポート出力にのみ都度反映する。
5. **上書き禁止**：BASTは外部ベンチマーク（相場観の参考）としてのみ
   使用し、financial_calc.pyが算定した内部判定区分（CF自走性・債務償還
   年数等のゾーン）を上書き・言い換えしない。合否判定は内部指標
   （特にCF自走性）が担う（§14）。

### Step 6：レポート組み立て

「全体構成（判定の3層構造）」節の順序で構成し、§10の出力2モード
（経営者向け／Grow3内部用）・停止条件を適用する。停止条件に該当する
場合は顧客向け出力を確定せず、小林レビューを要請する。

## 禁止事項

- 指標の算式・閾値・区分名称を本ファイルに複製すること（改訂時のズレの
  原因になる）。
- `judgment_status=screening_only`の結果に正式な区分語を割り当てて
  表示すること。
- 欠落した入力を`0`で埋めること、または`0`の入力を欠落として省略すること。
- BAST値を保存・キャッシュして次回以降に再利用すること、または取得不能時
  に記憶・推測で埋めること。
- 出力メタデータ（`ocf_source`等）が欠落した状態でレポートを作成すること。
