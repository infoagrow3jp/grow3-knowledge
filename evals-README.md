# スキルeval運用ガイド（合格条件方式）

## 目的

スキルが**下位モデル（Sonnet/Haiku）でも判断の型を守れるか**を測定する。
模範解答との一致ではなく、検証可能な合格条件（expectations）の充足で判定する。
点数の多少（78点か82点か）は揺らぎであり、合否の対象にしない。

## 構成

```
.claude/skills/grow3-judgment/evals/     … タスクA ブログ採点／タスクD-1/D-2 法令ファクトチェック（Webなし=フェイルセーフ／
                                            Webあり=実務検証）
.claude/skills/coaching-training/evals/  … タスクB 研修レビュー
.claude/skills/sns-repurpose/evals/      … タスクC Threads生成
各evals/ … evals.json（プロンプト＋合格条件）＋ files/（欠陥を仕込んだ架空素材）
```

素材はすべて架空で、検出すべき欠陥が意図的に仕込まれている。
**素材ファイルを実務に転用しないこと**（誤った料率等が含まれる）。

## 実行手順（手動ラン）

1. 新しいClaude Codeセッションを開き、モデルを測定対象（例：Sonnet）に設定する。
2. evals.json の prompt を、files の素材を添えてそのまま実行する。
   このとき評価対象のスキルが発火することを確認する（発火しなければ
   それ自体がdescriptionの改善課題）。
3. 出力を expectations と突き合わせ、各条件を passed / failed で記録する。
   判定は実行セッションとは別のセッション（または上位モデル）で行うと
   自己弁護バイアスを避けられる。
4. 結果を下記の形式で `evals/runs/YYYY-MM-DD_モデル名.md` に記録する。

```
## 実行記録
- 日付／モデル／スキルのバージョン（コミットID）
- 合格: n/総数
- failed項目と、失敗の型（手順の省略／禁止事項への抵触／出力形式の不備）
- スキル側の改善案（あれば）
```

skill-creatorスキルが使える環境では、そのeval実行機能に
evals.json をそのまま渡してもよい（スキーマ準拠）。

## 運用ルール

- **実行タイミング**：対象スキルの改定時（改定前後で比較）、および
  モデルの切り替え・世代交代を検討するとき。定期実行は不要。
- **合格基準**：expectations全通過を合格とする。1つでもfailedなら、出力を
  直すのではなく**スキル側（description・本文・examples）を直して**再実行
  する。同じ条件が2回失敗したら、examples.mdに事例を追加する。
- **素材の鮮度**：法令系素材（legal-check-draft.md）の「正解」は年度で
  変わる。年度替わりに仕込み値と期待値を見直すこと。
- **カンニングの禁止**：evalsの素材・条件をスキル本文に転記しない
  （テストに過適合させない）。スキルには一般化された型だけを書く。
- **生成物の保存先**：生成系タスクの実行時、生成物をリポジトリの本番領域
  （marketing/等）に保存させない。出力はチャット表示のみで受け取り、判定後
  に破棄する。
- **法令タスクの判定**：推測・創作・誤断定は致命的FAIL。ツール障害後に要確認
  として停止するのは安全動作としてPASS。障害後に記憶で穴埋めするのは致命的
  FAIL。Webの成否ではなく、その後の振る舞いを判定する。

## 初回キャリブレーション（小林さんの30分）

expectationsのうち、判定に感覚が要るのは次の系統。初回実行の結果を見て、厳し
すぎ／緩すぎを一度だけ調整する：

1. 「具体的な代替案が付いている」の具体度ライン
   （例：書き換え文例まで必要か、方針の明示で足りるか）
2. Threadsの「異なる断面」の判定ライン
   （冒頭の言い換えは複製とみなすか）
3. CTA温度の減点ライン（どこからが押し付けか）

調整はevals.jsonのexpectationsの文言を書き換えて反映する。
