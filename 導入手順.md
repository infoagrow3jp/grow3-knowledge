# Claude Code サブエージェント4体制 導入手順

## 構成

```
.claude/agents/training-reviewer.md   … 研修査読（タイプ判定→軸選択、修正しない）
.claude/agents/blog-reviewer.md       … ブログ5軸採点＋機械検出（AI語は警告扱い）
.claude/agents/compliance-checker.md  … 出荷前検品（スクリプト結果の解釈、haiku）
.claude/agents/legal-fact-checker.md  … 法定数値の一次情報検証（WebSearch+WebFetch）
.claude/settings.json                 … Stopフック（成果物完成時に検品を確実実行）
scripts/pre-delivery-check.sh         … 機械検品スクリプト（FAIL/WARN分離）
.githooks/pre-commit                  … コミット前の最終防衛線
```

設計原則：判断基準の正本はスキル側（grow3-judgment等）。各エージェントは
frontmatterの `skills:` でスキル本文を起動時にプリロードし、エージェント定義には
基準を複製しない。レビュアーは修正せず、指摘＋代替案のみ返す（第0原則）。

## 導入手順

1. **配置**：この一式をgrow3-knowledgeリポジトリのルートに展開する
   （既存の training-reviewer.md / blog-reviewer.md は上書き）。
2. **settings.jsonのマージ**：既に `.claude/settings.json` がある場合は、
   `hooks` ブロックだけを既存JSONに追記する（丸ごと上書きしない）。
3. **実行権限**：
   ```bash
   chmod +x scripts/pre-delivery-check.sh .githooks/pre-commit
   ```
4. **pre-commitの有効化**（リポジトリごとに1回）：
   ```bash
   git config core.hooksPath .githooks
   ```
   ※第2PCでも `git pull` 後に同コマンドを1回実行する。
5. **動作確認**：
   ```bash
   echo "テスト【要確認】" > /tmp/test.md
   bash scripts/pre-delivery-check.sh /tmp/test.md   # FAILが出れば正常
   ```
6. **前提**：grow3-judgment / coaching-training の両スキルが
   `.claude/skills/` に配置済みであること（skills:プリロードの参照先）。

## 検品の三重化（役割分担）

| 層 | 仕組み | タイミング | 性質 |
|---|---|---|---|
| 1 | Stopフック | Claude応答完了時 | 確実（LLMの判断に依存しない） |
| 2 | pre-commit | git commit時 | 確実（--no-verifyでのみ回避可） |
| 3 | compliance-checker | 納品前にproactive発火 | 結果の解釈と裁量判定を担当 |

スクリプトが機械判定（禁止事例・【要確認】・未確定表記＝FAIL、
色コード・読点・AI語＝WARN）、エージェントが裁量判定
（CPマスター準拠・視覚的ブランド適合・AI語の文脈判断）という分担。

**Stopフックで警告のまま終了した場合でも、未解消のFAIL項目がある限り
pre-commitでコミットは遮断される。警告が出た成果物は、コミット前に必ず
修正または意図的な除外判断（git commit --no-verify）を行うこと。**

## 運用メモ

- モデル配分：判断系（training/blog-reviewer）= inherit、
  検品 = haiku（安価・高速）、法令検証 = sonnet。
- legal-fact-checkerは検索結果のスニペットで確定せず、必ずWebFetchで
  原典を読む仕様。都道府県別料率は原則岡山県で確認。
- 禁止事例の検出パターン（「介護」×「離職率」共起等）は誤検知しうるが、
  FAILは「出荷不可」ではなく「人が確認するまで止める」意味なので許容する。
- 新しい検品項目はスクリプトに追記し、エージェント定義は触らない
  （基準の正本を増やさない）。

## 凍結前の確認（横断検査）

基準文書を凍結する前には、次の2種類の検査を行う。

- **縦の検査**：指定修正点が反映されているか、旧記述の残骸が残っていないか
- **横の検査**：改訂diff全体に、指示外の追加判断・例示・規則・値が混入していないか

凍結前には、指定修正点の残骸検査に加えて、基準版との差分全体を確認する。
指定対象外で追加・変更された値、例示、規則、設計判断がないかを検査し、
根拠のない追加を凍結によって既成事実化させない。

意図的に残す「未確定」表記は `scripts/pre-delivery-intentional-uncertainty.tsv`
に完全一致行として登録し、pre-commitではWARNとして監査可能にする。
未登録の「未確定」は凍結文書・DECISIONSであってもFAILとする。
